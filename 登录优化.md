这里的多租户隔离主要是指在这个 SaaS 级短链接系统中，如何保证不同用户（租户）之间的数据是完全隔离的。从源码中来看，主要通过以下几个维度实现多租户隔离以及用户注册和登录流程：

1. 如何实现多租户隔离？
在这个系统中，“租户”其实就是“用户（User）”。系统并没有做复杂的二级租户体系，而是通过以下两个核心机制实现了不同用户的数据隔离：

网关+拦截器级联提取用户身份（ThreadLocal 隔离）：
网关解析： 当用户请求到达网关（如 TokenValidateGatewayFilterFactory）时，网关负责校验 Token 是否合法。如果合法，网关会将解析出的 userId、username、realName 等信息通过 HTTP Header 向下传递给具体的微服务。
微服务拦截器（UserTransmitInterceptor）： 在微服务模块（如 project 或 admin）中，配置了 UserTransmitInterceptor 拦截器。它会从 Header 中读取 username 和 userId。
上下文存储（UserContext）： 拦截器读取到用户信息后，会将其存入 UserContext。UserContext 底层使用的是 java.lang.ThreadLocal。这意味着在当前请求的整个线程处理周期内，任何业务代码（如 Service 层处理短链接和分组时）都可以直接通过 UserContext.getUsername() 获取当前操作者，从而保证当前线程的操作严格限制在当前用户的数据范围内。
数据库查询级别隔离（username 或 gid 作为查询条件）：
核心业务表（如 t_link 短链接表、t_group 分组表）均包含 gid (短链接分组ID) 或 username 字段。
因为有了 UserContext，在执行增删改查时，底层查询和更新都会强制加上 username = 当前用户名 或对应的 gid 条件，从物理查询层面实现数据隔离，防止串号读取。
2. 用户注册流程是怎样的？
根据 UserServiceImpl.register() 的核心逻辑，注册流程为了保证高并发下的用户名唯一性和系统安全，经过了精细设计：

布隆过滤器拦截： 首先调用 hasUsername(username) 方法。这个方法底层使用的是 布隆过滤器（Bloom Filter） 快速判断该用户名是否已存在。如果布隆过滤器显示存在，直接抛出 ClientException(USER_NAME_EXIST)，快速拦截恶意注册并发。
获取分布式锁： 为了防止极端情况下的并发覆盖，基于 Redisson 获取分布式锁 LOCK_USER_REGISTER_KEY + 用户名。如果未能抢到锁，说明有别人正在注册这个名字，抛出已存在异常。
持久化到数据库： 将用户实体（UserDO）插入持久层。
初始化默认数据： 也就是调用 groupService.saveGroup(requestParam.getUsername(), "默认分组")。为了提升新用户的体验，一旦用户注册成功，系统会自动为该用户创建一个“默认分组”。
更新布隆过滤器： 将新注册成功的 username 写入到注册的布隆过滤器 userRegisterCachePenetrationBloomFilter 中，防范后续针对该用户的缓存穿透攻击。
兜底保障： 捕获数据库层面的唯一索引冲突异常 DuplicateKeyException 作为最后一道防线，确保无论如何不会出现同名用户。释放分布式锁。
3. 用户登录流程是怎样的？
根据 UserServiceImpl.login()，该系统采用的是 Token + Redis 的会话管理方案，并且处理了重复登录的逻辑：

校验账号密码： 首先根据传入的 username 和 password，以及 delFlag=0 查询数据库。如果不匹配，直接抛出异常提示“用户不存在或密码错误”。
判断是否已登录（踢出会话或续期）：
拿着 USER_LOGIN_KEY + username 去 Redis 中查询对应的 Hash 结构。
如果存在已登录的会话： 系统会给 Redis 中的这套 Token 重新续期（重置为 30 分钟过期），然后直接将之前的那一套 Token 返回给前端（也就是说，这个系统目前允许多端登录或者说在另一处登录时会复用同一个有效期 Token，并且刷新了它的活跃时间）。
生成全新登录态：
如果 Redis 中不存在该用户的登录信息，则通过 UUID.randomUUID().toString() 生成一个全新的 Token 标识（uuid）。
以 Hash 结构存入 Redis：
外层 Key：login_用户名
内层 Hash Key：token标识 (uuid)
内层 Hash Value：JSON化后的用户信息 (userDO)
设置有效期： 将该 Redis Key 强行设置 30 分钟的有效期（30L, TimeUnit.MINUTES）。
返回结果： 将生成的 uuid 作为 Token 返回给前端应用，前端后续每次请求只要带上这个 Token 即可通过网关校验。
