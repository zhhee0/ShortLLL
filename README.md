**. 基于自定义拦截器实现用户登录校验，结合 ThreadLocal 维护用户上下文，支撑多用户场景下的数据隔离与分组访问控制**

“这块我主要做的是统一认证和用户上下文管理。具体做法是，在请求入口通过自定义拦截器统一处理登录校验，先判断白名单接口，非白名单接口再从请求头中提取登录凭证并校验当前会话是否有效；认证通过后，把当前用户信息写入 ThreadLocal 封装的 UserContext，这样后续业务层就可以直接获取当前登录用户，不需要层层传参。请求完成后再清理上下文，避免线程复用带来的脏数据问题。基于这套机制，后续分组、短链、统计等核心业务都可以围绕当前用户做资源过滤和归属校验，从而支撑多用户场景下的数据隔离。”



**2. 使用 RabbitMQ 异步化访问统计写入，利用消息队列削峰填谷，保障高并发场景下的稳定落库；结合 Redis 实现消费端时间窗口幂等控制，降低重复消费导致的统计偏差**

“访问短链接时，我没有把 PV、UV、UIP、地区、浏览器、设备这些统计同步写库，而是先把统计数据封装成消息发到 RabbitMQ，再由消费者异步落库。这样跳转主链路只负责跳转和发消息，重的统计写库动作延后处理，消息队列在这里就起到了削峰填谷的作用，避免高并发下大量统计写请求直接打满数据库。消费端这边，我又加了一层 Redis 时间窗口幂等控制，每条统计消息都带一个业务生成的 messageId，消费者先用这个 messageId 去 Redis 做去重判断，只有第一次消费才继续执行业务，短时间内的重复消息会被拦住。这个方案不是金融级强幂等，但对访问统计这种最终一致场景是比较合适的，能有效降低重复消费带来的统计偏差。”



**3. 短链跳转场景下，采用布隆过滤器与空值缓存防止缓存穿透，缓存失效时结合分布式锁与双重检查机制重建缓存，避免缓存击穿**

“短链跳转是高频读场景，所以我在这条链路里做了三层保护。第一层是布隆过滤器，用来拦截那些从来没创建过的短链接请求，防止恶意穿透数据库；第二层是空值缓存，主要处理那些曾经存在但现在已经无效的短链，比如过期、回收站、删除这种场景，因为这些短链布隆过滤器通常还会判断成可能存在，所以要靠空值缓存拦住重复回源；第三层是分布式锁加双重检查，用来解决热点短链缓存失效时的并发回源问题。也就是缓存 miss 后不会所有线程都直接查库，而是先竞争锁，拿到锁的线程再做一次双重检查，确认缓存还是没恢复，才去查数据库并重建缓存，其他线程复用结果，从而避免缓存击穿。”



**4. 基于 Cache Aside 模式，采用先更新数据库、再删除缓存的策略，降低缓存与数据库不一致风险**

“我这里采用的是标准的 Cache Aside 思路：读请求先查缓存，没有再查数据库并回填；写请求不直接更新缓存，而是先更新数据库，再删除缓存。之所以选这个顺序，是因为删除缓存比更新缓存更简单，也更不容易在高并发下出错。如果采用先删缓存再更新数据库，中间很容易有读请求把旧数据重新写回缓存；而先更新数据库再删缓存，至少能保证数据库已经是新值，后续读请求回源时拿到的也是最新数据。这个模式在项目里主要用在短链接修改、移入回收站、回收站恢复这些场景，本质上都是业务状态先变，再让下一次读请求自动重建最新缓存。”

先更新数据库再删除缓存，删除后确实会让下一次读请求 miss，但这不一定就是缓存击穿。只有当被删的是热点 key，并且在缓存失效瞬间有大量并发请求同时回源数据库，才会形成击穿。在我这个项目里，短链跳转缓存失效后已经通过分布式锁和双重检查机制保护了重建过程，所以即使修改短链后主动删缓存，也只是有击穿风险，不会直接把数据库打穿。”

“删缓存会带来短暂 miss，热点场景下可能诱发击穿，但项目里用分布式锁和双重检查兜住了重建过程。”



**5. 使用 Redisson 分布式读写锁，支持高并发场景下对短链数据的同步修改**

“这块锁的重点不是泛泛地防并发，而是解决短链跨分组迁移和异步统计落库并发执行时的一致性问题。因为统计消费者在写数据前，需要先根据 fullShortUrl 查当前归属的 gid，而主表累计统计又是按 gid + fullShortUrl 更新的。如果这时候后台正好把短链从分组 A 迁到分组 B，就可能出现统计线程拿到旧 gid、后续把数据写到旧记录上的问题。所以我用 Redisson 分布式读写锁，以 fullShortUrl 作为锁粒度：统计消费者拿读锁，迁组操作拿写锁。这样多个统计读可以并发，但迁组写会独占，保证从读取归属到统计落库完成这段过程里，归属关系不会变化。这个方案的优点是能直接和当前数据模型对齐，缺点是锁范围偏大，后续如果要继续优化，我会优先从统计和归属解耦的角度入手。”

**6. 基于 Redis、AOP 和自定义注解实现用户、接口维度的限流能力，应用于短链创建、短链修改及后台统计查询等核心接口**

“我把限流做成了一套通用能力，而不是把规则写死在某个接口里。具体做法是先定义自定义限流注解，在接口上声明时间窗口、阈值和限流维度；然后通过 AOP 在方法执行前统一拦截，解析注解后根据当前用户、接口路径或者资源名生成限流 key。Redis 侧我用 ZSet 做滑动窗口，score 存请求时间戳，member 用时间戳加 UUID 保证唯一，再通过 Lua 脚本在 Redis 端原子完成窗口外数据清理、当前请求数统计、阈值判断和本次请求写入。这样既能避免高并发下统计和校验分离导致的竞态问题，也能把限流能力沉淀成统一组件。在这个项目里，我主要把它应用在短链创建、短链修改和后台统计查询这些核心接口上，分别从用户维度和接口维度去保护系统资源。”

7.点击一个短链 是如何跳转的？

比如 https://heng.beimxw.top/38z4A8。用户点击短链接后，后台先拿到这次请求对应的短链地址，先查 Redis 跳转缓存，命中就直接拿原始长链接做 302 重定向；如果缓存没命中，会先过布隆过滤器和空值缓存，拦截明显不存在或已经确认失效的短链请求。还不能确定时，再通过分布式锁保护回源过程，锁内做双重检查。确认确实需要查库后，系统会**先查短链路由表 t_link_goto，根据短链地址拿到对应的分组标识 gid**，再带着 gid + 短链地址 去查短链主表 t_link，校验这条短链是否存在、是否启用、是否过期。查到有效数据后，再把原始长链接写回 Redis，完成重定向；如果查到无效结果，就写空值缓存，避免后续继续穿透。与此同时，系统还会采集 PV、UV、UIP 以及设备、浏览器、地域等访问数据，并异步发到 MQ 落库。”

先查 Redis；没命中就走布隆过滤器、空值缓存和分布式锁；需要回源时，先查路由表拿 gid，再查主表校验短链状态；有效就重建缓存并重定向，无效就写空值缓存；最后把访问统计异步发 MQ。

8 短链生成

这个项目里的短链后缀生成算法用的是 MurmurHash32 + Base62。具体做法是，先把原始长链接和一个随机 UUID 拼接起来，避免同一个长链接每次都生成完全相同的候选值；然后对这个字符串做 MurmurHash32，得到一个 32 位整数；再把这个整数转成 Base62 字符串，也就是由数字、大小写字母组成的短码。生成候选短码后，系统还会做冲突校验，普通模式下先借助布隆过滤器快速判断是否可能已存在，加锁模式下会直接查数据库确认。如果发现冲突，就继续重新生成，最多重试 10 次。确认短码可用后，再写入短链主表和映射表，同时更新缓存和布隆过滤器。也就是说，这套方案本质上是‘哈希压缩 + Base62 编码 + 冲突重试’。

短链生成算法就是：长链接 + UUID 做 MurmurHash32，再转成 Base62，最后通过布隆过滤器或数据库做冲突校验，不行就重试

9网页输入用户名+密码，怎么保证安全地传给后端

登录成功以后，项目不会反复传密码，而是由后端生成一个 token，存到 Redis，会话默认 30 分钟并支持滑动续期。前端后续请求带的是 username + token，不是再带密码。所以密码只在登录这个动作里传一次，后面主要靠 token 做登录态校验。”真正保证传输安全的前提是部署时必须走 HTTPS。也就是说，浏览器到服务端这段链路要由 TLS 来保证加密传输，避免账号密码在网络里被明文窃听

10分表是怎么做的 什么可以当做分片键？

“这个项目里的分表是基于 ShardingSphere 做的，属于单库分表。逻辑表像 t_user、t_group、t_link、t_link_goto，底层都拆成了 16 张物理表，由 ShardingSphere 根据分片键自动路由。具体来说，t_user 和 t_group 是按 username 分表，因为这两张表天然围绕用户查询；t_link 是按 gid 分表，因为短链后台主要按分组管理；t_link_goto 是按短链地址分表，因为跳转入口最先就是按短链查映射。分片键一般要满足三个原则：高频查询会带上它、分布尽量均匀、并且尽量稳定。如果选了一个经常变的字段做分片键，就会带来数据迁移和一致性复杂度，这也是我这个项目里短链迁组场景会更复杂的原因。**ShardingSphere 再用 HASH_MOD 算法路由**。

- username 先参与哈希
- 再对分片数取模
- 最后决定落到 0~15 哪张表

11 不同用户对同一个url申请短链 结果应该是相同还是不同？ 如果设置为相同会有什么问题

在我这个项目里，同一个原始 URL 被不同用户申请短链，我会让结果是不同的。因为短链在这里不只是一个简单的长链接映射，它还带着明确的业务属性，比如属于哪个用户、哪个分组、有效期怎么设置、统计数据归谁、后续谁有权限修改或删除。所以我会把短链看成一个独立的业务对象，而不是单纯对原始 URL 做全局去重。

如果强行设计成相同，也就是不同用户对同一个 URL 永远复用同一个短链，看起来好像节省了存储，但会带来很多业务问题。第一是归属问题，这个短链到底算谁的；第二是权限问题，谁能修改、谁能删除；第三是统计问题，PV、UV、UIP 这些访问数据到底记到谁名下；第四是配置冲突，不同用户即使跳转到同一个原始 URL，也可能希望设置不同的有效期、备注、分组和运营策略。这样一来，短链就很难做用户隔离和后台管理。

所以在这个项目场景下，我会选择让不同用户申请同一个 URL 时生成不同短链，换取更清晰的归属、权限和统计边界。只有在那种纯公共短链服务、没有用户隔离和后台管理诉求的场景下，才可能考虑把同一个 URL 复用成同一个短链。”

12 **顺序性一般靠‘同 key 同队列 + 单消费者串行处理’来保证，但通常只能保证局部顺序，不保证全局顺序**

13分表下怎么扩容

分表扩容的本质是分片规则变化后的数据重分布问题。比如原来按 16 张表分片，扩到 32 张表后，很多历史数据按新规则已经不该留在原表里了，所以必须先建新表，再把旧数据迁到新分片，最后切换路由规则。小系统可以停机迁移，大一点的系统通常会采用双写、灰度切换和后台数据搬迁的方式平滑扩容。也就是说，分表扩容真正难的不是建表，而是如何在不停机或少停机的情况下完成规则切换和数据一致性保障。”

**分表扩容的核心不是加表，而是‘新规则上线前，怎么把旧数据平滑迁到新分布上**

- **短链跳转，缓存未命中：200 QPS 左右**
- **短链创建：100 QPS 左右**
- **后台统计查询：50 QPS 左右**
- **MQ 统计消费：200 条/秒左右**

短链跳转在缓存命中场景下大概能到 1200 QPS 左右，因为主链路主要就是 Redis 读取加 302 跳转，数据库基本不参与；如果缓存未命中，需要经过布隆过滤器、空值缓存、分布式锁再回源数据库，那吞吐会明显下降，大概在 200 QPS 左右。短链创建这种写接口，因为要生成短码、写主表和映射表、更新布隆过滤器，整体大概在 100 QPS 左右。后台统计查询因为涉及多表聚合和统计维度计算，通常大概在 50 QPS 左右。

所以这套系统在单机 2C4G 下，我会把它概括成‘跳转千级、写入百级、聚合几十级’。真正的瓶颈主要还是 MySQL，尤其是写入和聚合查询，其次是 Redis 在热点短链场景下的压力，以及单机 CPU、内存对 JVM 和连接池的限制。如果继续往上提性能，我会优先从提升缓存命中率、优化异步统计消费端落库成本开始，再往后才是数据库读写分离和应用多实例扩容。”

**15 布隆过滤器原理 短链存满了怎么扩容 占用多少空间**

布隆过滤器本质上是一个**位数组 + 多个哈希函数**。新增一个元素时，不是把元素本身存进去，而是先经过多个哈希函数计算出多个位置，然后把位数组里这些位置都置成 1。查询一个元素时，也走同样的多个哈希函数，如果对应位置里只要有一个是 0，就能确定这个元素一定不存在；如果这些位置全都是 1，就只能说明这个元素**可能存在**，不能 100% 保证一定存在。

**传统布隆过滤器本身不适合直接无损扩容。**

因为它初始化时就确定了：

- 位数组大小 m
- 哈希函数个数 k
- 预估数据量 n
- 误判率 p

如果短链数量持续增长，超过最初设计容量，布隆过滤器本身不会报满，但误判率会不断上升，所以工程上要么重建一个更大的布隆过滤器，把历史短链重新灌进去；要么做分层布隆过滤器，旧的保留、新的单独接入；再实际一点，就是一开始初始化时把容量预估得更大一些，尽量减少扩容频率。

至于空间占用，它和预估数据量、误判率直接相关。比如误判率按 1% 来算，100 万条短链大概只需要 1MB 多一点，1000 万条短链大概十几 MB，远小于直接把所有短链字符串完整存到 Redis Set 里的成本。所以它本质上是用很小的空间代价，换快速存在性判断能力。”

- 原理：位数组 + 多哈希
- 扩容：重建更大过滤器 or 分层过滤器
- 空间：百万级大约 1MB 左右，千万级十几 MB 左右（1%误判率）

“普通布隆过滤器一般不支持删除元素。因为一个元素经过多个哈希函数，会把位数组里的多个位置置成 1，但这些位置很可能同时也被别的元素共用。如果你删除某个元素时，直接把这些 bit 重新置成 0，就有可能把其他本来还存在的元素也误删掉，导致本来应该判断存在的元素被错判成不存在，这就出现了漏判。”

为什么是base62

Base64 我知道，但它不太适合直接拿来做短链后缀。因为标准 Base64 除了数字和大小写字母，还会包含 +、/ 和 =，这些字符放到 URL 路径里不够友好，往往还要额外转义或者改成 URL-safe 变种。Base62 则只包含数字和大小写字母，天然适合 URL，视觉上也更干净、更容易传播。所以短链场景下，通常会优先选 Base62，而不是标准 Base64。

16介绍下shardingSphere

“ShardingSphere 可以理解成一个数据库中间件，用来做分库分表、读写分离、数据加密这些能力。它的核心价值是对业务代码透明，应用层还是按逻辑表写 SQL，底层由它根据分片规则自动路由到具体库表。在我这个项目里，主要用它做单库分表和字段加密。比如 t_user 和 t_group 按 username 分表，t_link 按 gid 分表，t_link_goto 按短链地址分表；同时用户表里的手机号、邮箱字段也通过它做了加密。这样可以减少业务层自己维护分片路由的复杂度，但代价是分片键设计、跨分片查询和后续扩容会更复杂。”

**ShardingSphere 就是把分片、路由、加密这些数据库层能力统一收口，让应用层继续像操作逻辑表一样开发。”**

17shardingSphere 跨表 join操作的处理

ShardingSphere 对跨表 join 的处理，关键看两张表能不能被路由到同一个分片。如果两张表分片规则一致、分片键一致，并且 SQL 里带了分片键条件，那它可以把 join 下推到具体分片数据库本地执行，性能影响相对小；如果分片规则不同，或者查询里不带分片键，它就只能广播到多个分片执行，再把结果拉回来归并，性能会明显变差。所以在分库分表场景里，我一般不会默认复杂 join 是安全的，而是会优先从建模和分片设计上规避跨分片 join。比如让经常关联的表使用相同分片键，或者把一次复杂 join 拆成多次更可控的查询。”

**ShardingSphere 能做 join，但高不高效不取决于 join 本身，而取决于能不能精准路由到同一个分片。**

18 单表性能瓶颈，单表不能超过多少数据量   分页查询会出现的问题     深分页怎么解决的

单表没有一个绝对不能超过的固定数据量，但工程上通常在几百万到上千万级时就要开始关注性能瓶颈，尤其是索引维护、排序和分页查询都会明显变重。分页查询最典型的问题是 limit offset 深分页，页数越靠后，数据库需要扫描并跳过的数据越多，性能会越来越差。解决深分页最常见的方法是基于游标的书签分页，比如用上一页最后一条记录的 id 或 createTime + id 作为条件继续往后查，这样数据库可以直接走索引，不需要做大量无效跳过。如果业务必须支持页码跳转，也可以先查索引再回表，或者干脆限制最大可翻页数。我的理解是，深分页不是单纯 SQL 写法问题，本质上是大数据量下的扫描成本问题。

单表瓶颈通常出现在百万级以上；深分页的问题本质是 offset 越大，数据库浪费越大；最优先的解法是游标分页，而不是继续硬扛 limit offset

**Redis 里用一个 key 记录消费状态**

- key 前缀：short-link:idempotent:
- value：
  - "0" 表示处理中
  - "1" 表示已完成
- TTL：2 分钟

每条统计消息在生产时都会带一个业务侧生成的 messageId。消费者收到消息后，不会立刻落库，而是先拿这个 messageId 去 Redis 做一次占位。具体做法是用 setIfAbsent 写一个幂等 key，第一次消费会把它标记成处理中，也就是值设成 0，并设置 2 分钟 TTL；如果后续有重复消息进来，发现这个 key 已经存在，就说明这条消息要么已经处理完成，要么还在处理中。已完成的直接跳过，处理中则交给 MQ 重试。等业务真正落库成功后，再把 Redis 状态改成 1，表示已完成；如果业务中途失败，就删除这个幂等 key，让后续消息还能继续重试。这样就能在时间窗口内尽量保证同一条统计消息只会被真正消费一次，避免 PV、UV、UIP 被重复累计。”SET key 0 NX EX 120

如果让我重新设计这块，我会采用 Redis + Inbox 双层幂等方案。生产端先给每条消息分配全局唯一 messageId。消费端收到消息后，第一步先用 Redis 做快速占位，挡住短时间内并发重复投递的消息；第二步再把 messageId 写入本地消费记录表，也就是 Inbox，并对 messageId 建唯一索引。只有插入成功的线程才允许继续执行业务逻辑；如果插入失败，就说明这条消息已经处理过，直接返回。这样 Redis 解决高并发去重效率问题，数据库消费记录表解决最终持久化幂等问题。”







UV 的做法是先看请求里有没有 uv cookie，没有就生成一个 UUID 回写给客户端，然后用 Redis Set 按 fullShortUrl + uv 做判重，第一次访问就把 uvFirstFlag 标成 true；UIP 则是从请求里拿真实 IP，再用 Redis Set 按 fullShortUrl + ip 做判重，第一次访问就把 uipFirstFlag 标成 true。后面我会把 uv、uvFirstFlag 和 uipFirstFlag 一起封装进统计消息发到 MQ，消费者只根据这个首访标记去累计 UV 和 UIP，不再重复做判重。”

- UV：cookie + Redis Set 判重
- UIP：IP + Redis Set 判重
- 判重时机：**跳转主链路里先算好，再异步发 MQ**
