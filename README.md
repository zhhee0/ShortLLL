**. 基于自定义拦截器实现用户登录校验，结合 ThreadLocal 维护用户上下文，支撑多用户场景下的数据隔离与分组访问控制**

“这块我主要做的是统一认证和用户上下文管理。具体做法是，在请求入口通过自定义拦截器统一处理登录校验，先判断白名单接口，非白名单接口再从请求头中提取登录凭证并校验当前会话是否有效；认证通过后，把当前用户信息写入 ThreadLocal 封装的 UserContext，这样后续业务层就可以直接获取当前登录用户，不需要层层传参。请求完成后再清理上下文，避免线程复用带来的脏数据问题。基于这套机制，后续分组、短链、统计等核心业务都可以围绕当前用户做资源过滤和归属校验，从而支撑多用户场景下的数据隔离。”



**2. 使用 RabbitMQ 异步化访问统计写入，利用消息队列削峰填谷，保障高并发场景下的稳定落库；结合 Redis 实现消费端时间窗口幂等控制，降低重复消费导致的统计偏差**

“访问短链接时，我没有把 PV、UV、UIP、地区、浏览器、设备这些统计同步写库，而是先把统计数据封装成消息发到 RabbitMQ，再由消费者异步落库。这样跳转主链路只负责跳转和发消息，重的统计写库动作延后处理，消息队列在这里就起到了削峰填谷的作用，避免高并发下大量统计写请求直接打满数据库。消费端这边，我又加了一层 Redis 时间窗口幂等控制，每条统计消息都带一个业务生成的 messageId，消费者先用这个 messageId 去 Redis 做去重判断，只有第一次消费才继续执行业务，短时间内的重复消息会被拦住。这个方案不是金融级强幂等，但对访问统计这种最终一致场景是比较合适的，能有效降低重复消费带来的统计偏差。”



**3. 短链跳转场景下，采用布隆过滤器与空值缓存防止缓存穿透，缓存失效时结合分布式锁与双重检查机制重建缓存，避免缓存击穿**

“短链跳转是高频读场景，所以我在这条链路里做了三层保护。第一层是布隆过滤器，用来拦截那些从来没创建过的短链接请求，防止恶意穿透数据库；第二层是空值缓存，主要处理那些曾经存在但现在已经无效的短链，比如过期、回收站、删除这种场景，因为这些短链布隆过滤器通常还会判断成可能存在，所以要靠空值缓存拦住重复回源；第三层是分布式锁加双重检查，用来解决热点短链缓存失效时的并发回源问题。也就是缓存 miss 后不会所有线程都直接查库，而是先竞争锁，拿到锁的线程再做一次双重检查，确认缓存还是没恢复，才去查数据库并重建缓存，其他线程复用结果，从而避免缓存击穿。”



**4. 基于 Cache Aside 模式，采用先更新数据库、再删除缓存的策略，降低缓存与数据库不一致风险**

“我这里采用的是标准的 Cache Aside 思路：读请求先查缓存，没有再查数据库并回填；写请求不直接更新缓存，而是先更新数据库，再删除缓存。之所以选这个顺序，是因为删除缓存比更新缓存更简单，也更不容易在高并发下出错。如果采用先删缓存再更新数据库，中间很容易有读请求把旧数据重新写回缓存；而先更新数据库再删缓存，至少能保证数据库已经是新值，后续读请求回源时拿到的也是最新数据。这个模式在项目里主要用在短链接修改、移入回收站、回收站恢复这些场景，本质上都是业务状态先变，再让下一次读请求自动重建最新缓存。”



**5. 使用 Redisson 分布式读写锁，支持高并发场景下对短链数据的同步修改**

“这块锁的重点不是泛泛地防并发，而是解决短链跨分组迁移和异步统计落库并发执行时的一致性问题。因为统计消费者在写数据前，需要先根据 fullShortUrl 查当前归属的 gid，而主表累计统计又是按 gid + fullShortUrl 更新的。如果这时候后台正好把短链从分组 A 迁到分组 B，就可能出现统计线程拿到旧 gid、后续把数据写到旧记录上的问题。所以我用 Redisson 分布式读写锁，以 fullShortUrl 作为锁粒度：统计消费者拿读锁，迁组操作拿写锁。这样多个统计读可以并发，但迁组写会独占，保证从读取归属到统计落库完成这段过程里，归属关系不会变化。这个方案的优点是能直接和当前数据模型对齐，缺点是锁范围偏大，后续如果要继续优化，我会优先从统计和归属解耦的角度入手。”

**6. 基于 Redis、AOP 和自定义注解实现用户、接口维度的限流能力，应用于短链创建、短链修改及后台统计查询等核心接口**

“我把限流做成了一套通用能力，而不是把规则写死在某个接口里。具体做法是先定义自定义限流注解，在接口上声明时间窗口、阈值和限流维度；然后通过 AOP 在方法执行前统一拦截，解析注解后根据当前用户、接口路径或者资源名生成限流 key。Redis 侧我用 ZSet 做滑动窗口，score 存请求时间戳，member 用时间戳加 UUID 保证唯一，再通过 Lua 脚本在 Redis 端原子完成窗口外数据清理、当前请求数统计、阈值判断和本次请求写入。这样既能避免高并发下统计和校验分离导致的竞态问题，也能把限流能力沉淀成统一组件。在这个项目里，我主要把它应用在短链创建、短链修改和后台统计查询这些核心接口上，分别从用户维度和接口维度去保护系统资源。”

UV 的做法是先看请求里有没有 uv cookie，没有就生成一个 UUID 回写给客户端，然后用 Redis Set 按 fullShortUrl + uv 做判重，第一次访问就把 uvFirstFlag 标成 true；UIP 则是从请求里拿真实 IP，再用 Redis Set 按 fullShortUrl + ip 做判重，第一次访问就把 uipFirstFlag 标成 true。后面我会把 uv、uvFirstFlag 和 uipFirstFlag 一起封装进统计消息发到 MQ，消费者只根据这个首访标记去累计 UV 和 UIP，不再重复做判重。”

- UV：cookie + Redis Set 判重
- UIP：IP + Redis Set 判重
- 判重时机：**跳转主链路里先算好，再异步发 MQ**

你这个项目里 UV 和 UIP 都是按 **短链维度** 存 Redis Set 的：

- short-link:stats:uv:{fullShortUrl}
- short-link:stats:uip:{fullShortUrl}

是的，按短链维度用 Set 判重在超热点短链场景下会有大 key 风险，当前是用简单实现换开发效率，后续可以通过按天拆 key 或用 HyperLogLog 优化。

“Cookie 可以理解成浏览器端按域名保存的一组键值对，不是只能存一个值，而是可以存多个独立字段，比如 uv、token、username 都可以是不同的 Cookie。它当然也有有效期，如果不设置就是会话级，浏览器关闭后可能失效；如果设置了 Max-Age 或 Expires，就可以持久化保存一段时间。像我这个项目里的 uv，本质上就是一个轻量的匿名访客标识，用来做 UV 判重，比较适合放在 Cookie 里。”
