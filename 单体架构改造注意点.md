# 面试防坑指南：如果项目包装成单体架构，我还需要注意哪些逻辑变化？

如果你决定在简历上把该项目包装成**单体架构项目（部署多节点集群）**，除了前文提到的“网关消失”、“Feign 调用变成 @Autowired”、“登录拦截变 Spring MVC 拦截器”以外，还需要小心面试官针对“单体”这一设定可能抛出的一些业务连环问。

为了防止穿帮或逻辑不自洽，请务必在脑海中理清以下短链核心逻辑的变与不变：

---

## ⭐️ 1. 绝对**不改变**的逻辑（面试重点关注）

很多初学者容易觉得“只要是单体系统，就不需要用到分布式组件了”。这是一个**巨大的误区**。单体系统通常也是要**集群部署**（比如起 3 个相同的 JAR 包实例挂在 Nginx 后面）来抗并发的。

所以，以下由于“多节点运行”或“高并发要求”而引入的技术，**即使在单体架构下，逻辑也完全不变，照说不误！**

### 1.1 短链跳转的异步统计必须要有 (RabbitMQ)
*   **面试官可能会问**：“既然是单体项目，直接用线程池去查数据库写统计日志不行吗？为什么还要用 RabbitMQ？”
*   **你的回答基调（不变）**：“即使是单体程序，短链重定向（302 跳转）接口的并发依然极高（系统的核心 QPS）。如果在跳转的同时直接写数据库去统计 PV、UV、UIP，即使是用线程池，当流量洪峰到来时也会瞬间打满数据库连接池，导致整个服务宕机。因此，引入 RabbitMQ 的目的是为了『流量削峰』和『异步解耦』。用户请求到达后，我只需把统计数据扔进 MQ 就能立刻返回 302 给用户，后台有个专有的消费者线程慢慢从 MQ 里捞数据往数据库里落盘保护数据库，这点和微服务是一致的。”
*   **区别仅仅在于**：MQ 的生产者（抛数据）和消费者（存数据）代码，现在写在同一个工程代码域的不同包里了，而不是跨网络的服务。

### 1.2 防止短链重复生成的锁必须要有 (Redisson)
*   **面试官可能会问**：“单体架构下，防止并发生成同一个短链时，你在代码里加个 `synchronized` 不就行了吗，干嘛用 Redis 分布式锁？”
*   **你的回答基调（不变）**：“如果在简历里，该单体项目是集群部署的（多个 JVM 进程运行在不同机器上）。`synchronized` 只能锁住当前这一台机器的 JVM。当高并发请求打到 Nginx，被分发到两台不同的服务器上时，依然可能绕过同一个布隆判重的逻辑，双双向数据库插入同样的短链接（违反唯一索引要求）。因此，必须使用基于 Redis 的分布式锁（`Redisson`），保证全局唯一性。”

### 1.3 缓存和布隆过滤器必须要有 (Redis / Guava)
*   布隆过滤器（防止短链误判缓存穿透）、空值缓存等逻辑，只要有频繁的数据库读取操作，不管是不是微服务，都是必需的性能优化手段，机制完全一致。

## ⚡️ 2. 会发生**改变**的逻辑（需要小心规避）

以下是你必须要修改话术的地方：

### 2.1 数据库事务 (Transaction) ：分布式变本地
*   **原微服务架构**：如果一个业务涉及修改“管理员”（Admin 服务所在的数据库）还要修改“短链接”（Project 服务所在的表）。在微服务下属于**分布式事务**问题（通常会扯到 Seata 或者基于 MQ 的最终一致性）。
*   **单体架构下**：千万别再提 Seata！在单体架构下，所有的数据库表极大概率处于同一个物理数据库（或分库分表的同一套逻辑连接池里）。所有的事务都是**本地事务**。
*   **话术改变**：“所有对多张表的级联操作（比如创建短链接时既要保存短链表，又要更新该用户该分组的数量统计），我都直接包裹在 Spring 的 `@Transactional` 注解内，一旦其中某张表保存失败抛出异常，整个事务自动回滚，这就大大减轻了分布式状态不一致的可能。”

### 2.2 路由聚合 (Gateway) ：全部由前端/Nginx代理
*   **原微服务架构**：前端向 `/api/short-link/admin/**` 发请求，网关 (Spring Cloud Gateway) 取出识别 `admin` 向用户微服务转发。
*   **单体架构下**：没有网关的事了。
*   **话术改变**：“前端不管调用的是用户的接口还是短链接的接口，在单体下，域名和端口都是一致的。前端直接访问我这个完整的工程应用所在的 Tomcat 即可。如果我们配置了多台单体机器扛并发，前端请求会先打到 Nginx 服务器上，由 Nginx 进行简单的按权重轮询式的负载均衡，将流量打给背后的 Web 服务节点。”

### 2.3 分布式链路追踪 (SkyWalking/Zipkin) ：可有可无
*   如果在微服务版里写了用到类似 SkyWalking 等链路追踪监控。在单体里尽量别提这个了或者淡化，因为单个服务里哪行代码慢直接看本地日志就一清二楚了，没有“追踪跨服务究竟是谁挂了”这个强需求。

## 📝 面试实战总结表

| 考察点 | 微服务架构（真实情况） | 单体架构（你的面试包装） | 当被面试官质疑时如何解释？ |
| :--- | :--- | :--- | :--- |
| **模块划分** | **物理拆分**：跑在不同的端口、JVM乃至服务器上 | **逻辑拆分**：跑在同一个端口的JVM上，通过不同的 maven module 或 package (包名) 隔离。 | 就算没分微服务，代码结构也是按高度解耦的DDD/业务模块划分的，互相通过 `@Autowired` 而非 `Feign` 调用。 |
| **核心业务防重** | Redisson 分布式锁 | Redisson 分布式锁 | 虽是单体也是多机高可用部署(`synchronized` 无效)，锁依然得用分布式的。 |
| **异步统计** | RabbitMQ (跨服务发消息) | RabbitMQ (同服务发消息) | 保护核心的重定向短跳不因为写库超时阻断，所以同样需要消息中间件削峰解耦。 |
| **登录鉴权** | Gateway 网关强校验 + Feign Header 透传向下 | Spring MVC Filter/Interceptor 强校验 + 存入 ThreadLocal 随用随取 | 没有了物理边界隔离，减少了跨网络开销，可以直接基于内存获取当前上下文(`UserContext`)。 |
| **数据一致性** | 复杂的微服务分布式事务保障 | 直接粗暴好用的 `@Transactional` 本地事务控制 | 如果不分库，一个事务全部包裹就行了，这就是单体的直接红利。 |
