# 短链接项目（单体架构版）面试指南：登录与拦截设计

有时在面试中，为了避免被引入复杂的分布式组件问题（如服务治理、分布式事务、微服务鉴权等），将项目包装为**单体架构 (Monolithic Architecture)** 是非常明智的策略。

如果在单体架构下，项目中原有的 **Gateway (网关)**、**Admin (管理后台)**、**Project (核心业务)** 三个模块将合并为一个运行在单一进程中的 Spring Boot 应用程序。

以下是在面试中讲解“单体架构下用户登录与拦截鉴权”的标准设计与话术。

---

## 1. 单体架构移除了哪些复杂的微服务组件？

在单体架构下，你需要向面试官明确表示不需要以下组件：
*   ❌ **移除 Spring Cloud Gateway**：因为所有 Controller 都在一起了，不需要网关来进行统一路由和第一层鉴权。
*   ❌ **移除 OpenFeign 及其拦截器**：后台业务代码可以直接通过 `@Autowired` 互相调用对应 Service 层代码，不需要发 HTTP 远程调用，也不再存在复杂的“Header 跨服务透传用户信息”的问题。

---

## 2. 登录流程设计 (Login Flow)

登录的核心本质没有变，依然是通过 Redis 生成并保存 Token。

1.  **前端请求**：用户输入账号密码，前端发起登录 POST 请求。
2.  **账号校验**：Controller 接收请求，调用 Service 层查询数据库校验密码。
3.  **生成 Token 与存入 Redis**：
    *   校验通过后，生成一个 UUID 字符串充当 Token。
    *   以 `short-link:login:{username}` 为 Redis Key，以 UUID 为 HashKey，将用户的基本信息序列化为 JSON 字符串存入 Hash 的 Value 中。
    *   为该 Key 设置 **30 分钟的有效期**（如果在续期方面想讲得严谨一点，可以在后续请求拦截器里顺延这个时间）。
4.  **下发 Token**：将这个 UUID Token 返回给前端，前端将其持久化在基于会话的 Cookie 中。

---

## 3. 全局鉴权与拦截设计 (Interceptor & Context)

这是单体架构与微服务差异最大的地方。由于没有了网关，鉴权防线退回到 Spring MVC 自身。我们采用 **Spring MVC 拦截器 (HandlerInterceptor)** 结合 **ThreadLocal** 来完成。

### 3.1 拦截器 (LoginInterceptor) 工作流程

在项目中定义一个全局拦截器 `LoginInterceptor`，实现 `HandlerInterceptor` 接口：

1.  **拦截路径拦截与放行**：
    *   配置拦截规则：涵盖所有业务 API `/api/**`。
    *   配置白名单：排除登录、注册等不需要 Token 的 URL。
2.  **`preHandle` 获取参数**：
    *   请求到达时，从 `HttpServletRequest` 的 Header（或 Cookie）中拿到 `username` 和 `token`。
3.  **Redis 查验身份**：
    *   使用这两个参数去 Redis 查找是否存在对应的 Hash 记录。
    *   **失败**：如果在 Redis 读取不到，说明未登录或过期，直接拦截请求，向前端硬性返回 HTTP `401 Unauthorized`。
4.  **上下文绑定 (重点)**：
    *   **成功**：如果在 Redis 取到了用户信息 JSON，将其反序列化为 `UserInfoDTO`。
    *   调用 `UserContext.setUser(userInfoDTO)`，**把用户信息直接绑定到当前处理该请求的线程 ThreadLocal 中**。
5.  **`afterCompletion` 安全清理 (关键得分点)**：
    *   当请求执行完毕并且视图渲染完成后（即便发生异常也会执行），从 `afterCompletion` 方法中强制调用 `UserContext.removeUser()` 清理当前线程的 ThreadLocal 数据。
    *   *面试话术*：“因为 Tomcat 等 Web 容器使用的是线程池，线程会被复用。如果请求处理结束后不清理 ThreadLocal，上一个请求的用户数据就会残留，导致下一个无关联的请求进来时发生严重的‘权限越权’和‘上下文串车’的 Bug，也极易导致内存泄漏。”

### 3.2 随处可用的 UserContext

由于拦截器已经在请求刚进来最外层做好了 ThreadLocal 的绑定，这使得同一条线程下的所有后续操作都在同一个进程内。

这带来了绝佳的开发体验：
在你的 Controller、Service 甚至 MyBatis-Plus 的自动填充拦截器里，任何想知道“当前是谁在操作”的地方，都不再需要往方法里冗余传入 `UserToken` 参数，直接一句话：
```java
String username = UserContext.getUsername();
String userId = UserContext.getUserId();
```
就能获取到用户身份执行业务逻辑。

## 4. 面试可用话术总结

如果要用一段话总结，你可以对面试官这样说：

> “在单体架构下，用户的登录与鉴权体系非常内聚。
> 登录时，后端生成 UUID 的 Token 并把用户信息缓存在 Redis 中设置过期时间，前端保存在 Cookie 内。
> 在拦截层面，我放弃了重量级的网关，而是使用 **Spring MVC 的全局拦截器 (HandlerInterceptor)** 统一拦截除白名单外的所有接口。
> 拦截器不仅去 Redis 校验 Token 的合法性保障系统安全，还在校验通过后，利用 **ThreadLocal** 将解析出的用户信息绑定到当前请求的上下文 (`UserContext`) 中。这样一来，整个请求从 Controller 到 Service，都不需要再经过网络层或者繁琐的参数传递，随叫随取。同时，我也在拦截器的最后环节做好了 `remove` 清理工作，防止线程池复用导致的内存泄漏和数据污染问题。”

## 5. 补充：单体架构下还需要分模块吗？是如何运行调用的？

在微服务架构中，`Admin` 和 `Project` 是两个**独立启动、独立部署**的进程（各有各的启动类、各有各的端口）。但在单体架构下，**它们都会合并到一个启动类（一个 JVM 进程）中**运行。

### 5.1 还需要分用户 (Admin) 和 Project 吗？
**答案是：需要分，但拆分的方式变成了“逻辑拆分”（包结构 / Maven 模块层面），而不是物理微服务拆分。**

即使是单体架构，为了代码不变成“屎山”，依然需要遵循“高内聚、低耦合”的设计：
### 5.2 单体架构的包结构（逻辑拆分示例）

在单体应用里，我们推荐使用**按业务功能分包 (Package by Feature)**，或者在顶层做模块化分包。以下是一个典型的单体架构合并后的包结构设计思路：

```text
com.nageoffer.shortlink
├── ShortLinkApplication.java       // 整个项目的唯一 Spring Boot 启动类
├── common                          // 全局公共组件 (原属各个微服务的 common 合并)
│   ├── config                      // 全局配置 (MyBatis, Redis, WebMvc拦截器等)
│   ├── constant                    // 全局常量
│   ├── convention                  // 全局规约 (统一返回结果 Result, 异常处理等)
│   ├── enums                       // 全局枚举
│   └── util                        // 工具类
├── admin                           // ★ 逻辑模块 1：后台管理模块 (原 Admin 微服务)
│   ├── controller                  // 用户、分组相关的 Controller
│   ├── service                     // UserServiceImpl, GroupServiceImpl
│   ├── dao                         // 
│   │   ├── entity                  // UserDO, GroupDO
│   │   └── mapper                  // UserMapper, GroupMapper
│   └── dto                         // Req 和 Resp 下的请求响应对象
└── project                         // ★ 逻辑模块 2：短链接核心模块 (原 Project 微服务)
    ├── controller                  // 短链生成、跳转、统计的 Controller
    ├── service                     // ShortLinkServiceImpl, RecycleBinServiceImpl 
    ├── dao                         // 
    │   ├── entity                  // ShortLinkDO, LinkAccessLogsDO 等
    │   └── mapper                  // ShortLinkMapper
    ├── dto                         // 短链业务下的请求和响应实体
    └── mq                          // 消息队列相关 (消费者和生产者)
```

**这种包结构的好处：**
1. **边界清晰**：`admin` 只处理用户信息和后台配置，`project` 专注于短链接映射转跳和监控，井水不犯河水。如果未来业务庞大到必须拆分微服务，只需把 `admin` 和 `project` 这两个包各自独立拎出去建新的 Maven 项目即可。
2. **极简调用**：在微服务版本中，`admin.controller` 需要通过 Feign 调用 `project`；在上述单体结构里，`admin.controller` 只需要写一行 `@Autowired ShortLinkService shortLinkService;` 即可发起同内存下的光速调用。
3. **消除冗余**：原本分散在各个微服务里的相同工具类、配置类、异常拦截器等，都在 `common` 包里完成了大一统，去掉了大量的重复代码。

### 5.3 单体架构是如何运行的？
单体架构最大的优势在于**消灭了网络延迟和复杂的分布式调用**：
* **单点启动**：如上面的结构所示，项目只有一个 `ShortLinkApplication.java`。启动后，只会占用一个端口（例如 8080）。前端不管是发请求登录、还是去查询短链接，全都是打到这同一个网址的相同端口上。
* **内存级的方法调用 (彻底抛弃 OpenFeign)**：
  在微服务里，`Admin` 模块想查某个分组下有多少短链接，必须用 **OpenFeign** 发送一次网络 HTTP 请求给 `Project` 模块。如果中间网络抖动，还会超时。
  **但在单体架构中**，这两个模块运行在同一个 JVM 进程里。`Admin` 里的 `GroupServiceImpl` 想知道短链接数量，只需要通过 `@Autowired` 直接注入 `project` 包下的 `ShortLinkService`，然后像调用普通的 Java 方法一样强类型调用过去。
  这就把一次**毫秒级的网络开销**，变成了一次**纳秒级的内存函数指针跳转**，性能和调试体验都大幅提升。
