# 短链接的回收站与删除机制详解

在本项目中，短链接的删除流程分为两步：“移入回收站” 和 “从回收站彻底删除”。

无论进行哪一步，**在数据库层面都没有执行物理删除（`DELETE` 语句）**。系统的底层设计完全采用的是**逻辑删除（软删除）**。

下面我们对应到具体的代码和数据库字段来看看它是怎么流转的。

---

## 一、 第一步：移入回收站
当你在短链管理页面点击“删除（移入回收站）”时，后端调用的是 `RecycleBinServiceImpl.saveRecycleBin` 方法。

### 1. 改变的字段：`enable_status`
移入回收站的本质是修改了短链接的**启用状态**。
*   **修改前**：`enable_status = 0` （正常可见）且 `del_flag = 0`
*   **修改后**：**`enable_status = 1`** （表示被停用/进入回收站）且 `del_flag = 0`（尚未被彻底删除）

### 2. 附带操作：清理缓存
不仅修改了数据库状态，代码里还执行了一步关键操作：
```java
stringRedisTemplate.delete(String.format(GOTO_SHORT_LINK_KEY, requestParam.getFullShortUrl()));
```
这会把目标短链在 Redis 里的缓存（控制跳转的那个 Key）**立刻删掉**。
这样一来，就算别人手里拿到了这个短链接去浏览器里访问，也会因为它已经在回收站里（`enable_status = 1`）而无法成功跳转，直接引导到系统的 404/失效提示页。

---

## 二、 第二步：从回收站中彻底删除
如果在回收站列表中再次点击“删除”，后端调用的是 `RecycleBinServiceImpl.removeRecycleBin` 方法。

这表示用户希望**彻底且永久地抛弃这根短链接**。

### 1. 改变的字段：`del_flag` 和 `del_time`
这一次，代码触及了真正的“逻辑删除”防线：
*   **修改前**：`enable_status = 1` 且 `del_flag = 0`
*   **修改后**：**`del_flag = 1`** （被逻辑删除）并且更新了 **`del_time = 当前时间的时间戳`**。

### 2. 是永久物理删除吗？
**绝不是永久物理删除（不是 `DELETE FROM t_link`）。**

*   **对用户来说**：它是“永久删除”的。因为项目中所有的正常查询（比如分页查短链、查回收站）底层都会拼接 `.eq(del_flag, 0)` 的条件，只要 `del_flag` 变成了 1，这条记录在业务系统里就“灰飞烟灭”了，既不可能在列表里看到，也不可能被恢复。
*   **对数据库来说**：这条记录**依然安安静静地躺在硬盘里**。

### 3. 灵魂拷问：如果不加 `del_time`，只用 `[gid, full_short_url, del_flag]` 做唯一索引不行吗？

**答案是：绝对不行！如果在唯一索引里只用 `del_flag`，那么同一个短链接，你这辈子只能删除一次。**

我们可以推演一下真实的灾难场景：
假设联合唯一索引是 `(gid, full_short_url, del_flag)`。

1. **第一次创建**：你在这个分组下建了一个短链 `a.link/xyz`。
   * 此时数据库记录：`del_flag = 0`。
   * **当前索引池**：`('group1', 'a.link/xyz', 0)`。
2. **第一次删除**：你把这个短链移入回收站并彻底删除了。
   * 此时数据库记录更新为：`del_flag = 1`。
   * **当前索引池**：`('group1', 'a.link/xyz', 1)`。
3. **第二次创建**：过了一阵子，你又想建一个完全一样的短链 `a.link/xyz`。
   * 没问题，插入新记录 `del_flag = 0`。
   * **当前索引池**同时存在两条数据：`('group1', 'a.link/xyz', 1)` 和 `('group1', 'a.link/xyz', 0)`。目前一切正常。
4. **👉 灾难发生（第二次删除）**：现在，你想把这个新建的同名短链也删掉！
   * 系统尝试将它的 `del_flag` 也更新为 `1`。
   * 但此时，索引池里必须要存入 `('group1', 'a.link/xyz', 1)`。
   * **轰！唯一键冲突报错 (DuplicateKeyException)！** 因为在第 2 步的时候，索引池里已经存在一把一模一样的钥匙了！

**破局之道：引入时间戳 `del_time`**

为了解决这个“只能删一次”的死局，项目中巧妙地把唯一索引设计成了 `[gid, full_short_url, del_time]`：
*   **正常存活的短链**：`del_time = 0`。无论你怎么建，只要还活着，它的索引就是 `('group1', 'a.link/xyz', 0)`。
*   **被逻辑删除的短链**：`del_time = 真实的秒级时间戳`。
    * 第一次删除：`del_time = 1710000001`。
    * 第二次删除：`del_time = 1710000088`。

因为时间是在不断流逝的，每一次删除得到的时间戳都绝对**各不相同**！这样就完美化解了软删除下的唯一索引冲突问题。

---

## 面试总结话术
> "在我的项目中，短链接的删除经历了两阶段流转，全程采用**软删除（逻辑删除）**机制，不执行任何物理删除。
> 
> 第一阶段是**移入回收站**：此时只修改 `enable_status` 字段由 0 变为 1，同时删除对应的 Redis 跳转缓存，切断外网用户的访问。用户还可以随时在回收站里把它恢复（改回 `enable_status = 0`）。
> 
> 第二阶段是**回收站内彻底删除**：此时会修改 `del_flag = 1`，并同步更新 `del_time` 为当前时间戳。从业务视角来看，这根短链彻底消失不可恢复；从底层视角来看，数据依然保留在库中，作为日后的数据审计和溯源凭证。之所以要记录 `del_time`，是为了配合数据库级别的联合唯一索引，防止这条被软删除的记录卡死未来同名短链的再次创建。"
