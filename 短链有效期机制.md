# 短链接有效期机制详解

在面试中，如果被问到短链接的“有效期控制”和“失效处理”在代码层面是如何实现的，你可以从以下几个维度进行严密且有层次的回答。

---

## 1. 修改短链接过期时间是怎么做到的？

修改过期时间的本质是：**修改数据库元数据 + 缓存有效期控制**。

### 1.1 数据库层面的修改
当用户在后台编辑短链接并修改有效期时，前端会发送包含最新 `validDateType`（有效期类型：永久/自定义）和 `validDate`（具体的过期时间）的请求。

在代码实现 `ShortLinkServiceImpl.updateShortLink` 中：
*   **组装更新逻辑**：使用 MyBatis-Plus 构造 `LambdaUpdateWrapper`。
*   **永久有效处理**：如果用户选择的是“永久有效”（`VailDateTypeEnum.PERMANENT`），代码会强制将数据库表 `t_link` 中的 `valid_date` 字段**更新为 `null`**。
*   **具体时间有效**：如果选择了自定义时间，则将传入的 `validDate` （如 `2025-10-31 23:59:59`）直接覆盖更新到数据库表中。

---

## 2. “已失效”在代码层面是如何判断与流转的？

短链接的“失效”并非是通过定时任务（如 XXL-Job）去后台悄悄把状态改成“已失效”的（那样对数据库压力极大且有延迟），而是采用**“惰性判断 + 缓存 TTL TTL 自动过期”**的方式。

具体的拦截判断发生在用户访问短链接重定向的接口 `ShortLinkServiceImpl.restoreUrl` 中：

### 2.1 缓存维度的失效机制（第一道防线）
当一个正常的短链接被首次访问并加载到 Redis 时，会利用 `LinkUtil.getLinkCacheValidTime(validDate)` 计算当前时间距离过期时间还有多少毫秒。
*   然后将这个计算出的毫秒数，作为 Redis Key (`goto:%s`) 的 **TTL (存活时间)**。
*   **生效过程**：一旦时间到达设置的过期点，Redis 底层会自动将这个 Key 删除跳。所以，**从缓存层面看，Key 不存在了，就意味着短链接可能失效了**。

### 2.2 数据库维度的懒加载拦截（第二道防线）
当短链在 Redis 中过期被淘汰后，随后的第一次访问虽然经过了布隆过滤器的拦截（因为在布隆过滤器看来该短链曾经过合法创建，依然判定存在），但不可避免地会去查一次数据库。

这就触发了代码中最核心的失效判断逻辑片段：
```java
ShortLinkDO shortLinkDO = baseMapper.selectOne(queryWrapper);

// ★ 核心判断：如果查不到，或者 (有限期类型 且 数据库中的有效时间在这个瞬间 < 当前最新时间)
if (shortLinkDO == null 
    || (shortLinkDO.getValidDate() != null && shortLinkDO.getValidDate().before(new Date()))) {
    
    // 发现确实过期了！防止别人疯狂刷这个过期的短链导致缓存穿透
    // 在 Redis 中立即存入一个空值 ("-")，并给它 30 分钟的保护期
    stringRedisTemplate.opsForValue().set(
        String.format(GOTO_IS_NULL_SHORT_LINK_KEY, fullShortUrl), 
        "-", 30, TimeUnit.MINUTES);
    
    // HTTP 302 重定向到前端的“页面不存在/已失效”的 404 引导页
    ((HttpServletResponse) response).sendRedirect("/page/notfound");
    return;
}
```

### 2.3 面试话术提炼

> “在短链接的过期管控上，我采用的是**惰性失效加缓存自动清理**的策略，避免了引入繁重的分布式定时任务。
> 
> 在修改失效时间时，我主要是更新数据库 `t_link` 表的 `valid_date` 字段。
> 而在真实的跳转接口中，所谓的『已失效』体现在两个层面：
> 1.  如果短链接尚未加载到缓存，或者已经在 Redis 中过了 TTL 生存期被自动删除了，请求会打到数据库；
> 2.  从数据库查出记录后，立刻做一层判断：`validDate` 不是 null 且它的小于 `new Date()`。只要命中这条规则，我就不仅将其判定为『已失效』并重定向到系统的 `notfound` 友好提示页，**同时由于它确实存在于布隆过滤器中，为了防止接下来的高并发恶意攻击导致缓存穿透，我还会往 Redis 里存入一个长达 30 分钟的空对象 (`-`) 来阻挡流量**。
> 
> 这样既保证了业务严谨，也保障了高并发下的数据库安全。”
