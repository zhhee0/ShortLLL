# 项目漏洞与设计缺陷分析报告

经过对短链接项目（ShortLink）核心代码的走查，发现该项目在安全设计、权限控制和代码健壮性上存在若干明显的漏洞和缺陷。以下是主要问题的汇总：

---

## 一、 严重安全漏洞

### 1. 密码明文存储与比对（CWE-256 / CWE-319）
**位置**：`UserServiceImpl.java` -> `register` / `login`
**缺陷描述**：
- 用户注册时，前端传来的密码没有经过任何哈希处理（如 BCrypt、PBKDF2 等），甚至连最基础的 MD5 加盐都没做，直接以明文形式入库 (`t_user` 表)。
- 登录逻辑中，直接使用明文进行比对：`.eq(UserDO::getPassword, requestParam.getPassword())`。
**风险**：一旦数据库发生泄露（拖库、SQL注入等），所有用户的账号密码将直接暴露，面临极高的撞库风险。这是现代 Web 开发中绝对不可接受的红线漏洞。

### 2. 内部服务身份防伪造缺陷（网关绕过 / Header Spoofing）
**位置**：`UserTransmitFilter.java` & `TokenValidateGatewayFilterFactory.java`
**缺陷描述**：
- 后端服务（如 Admin 模块）在 `UserTransmitFilter` 中，直接从 HTTP 请求头中读取 `username`、`userId` 等字段来构建当前用户上下文（`UserContext`）。
- 虽然 Spring Cloud Gateway 在鉴权成功后会可靠地重写这些 Header，但如果**内网隔离防护不当**（例如 Admin 模块的内网端口直接对公网暴露，或者 Nginx 转发规则漏洞），攻击者可以直接跳过网关，向底层微服务发送带有伪造 `username` 和 `userId` Header 的 HTTP 请求。
**风险**：攻击者可以轻易伪造系统中的任何用户身份（甚至管理员），从而越权接管其他用户的短链接资产。

### 3. 水平越权与数据隐私泄露（IDOR）
**位置**：`UserController.java`
**缺陷描述**：
- 大量数据查询接口缺乏"数据所有权"校验。例如 `/api/short-link/admin/v1/user/{username}` 接口（`getUserByUsername`），底层仅执行 `select * from t_user where username = ?`。
- 代码中**完全没有校验**当前登录的身份（`UserContext.getUsername()`）是否与被查询的 `{username}` 变量一致。
- 更夸张的是 `/api/short-link/admin/v1/user/list-usernames` 接口（`listUsernames`），居然通过全表扫描（即便只 select username）直接暴露了系统里所有的用户名。
**风险**：只要注册了一个正常账号（有合法 Token 经过网关），就能任意枚举、拉取系统中所有其他用户的用户名、真实姓名等个人隐私数据（`UserActualRespDTO` 更是没有脱敏的实体）。

## 二、 架构与代码健壮性缺陷

### 4. 接口入参“零”校验（Missing Input Validation）
**位置**：全局 Controller 层及 DTO 对象
**缺陷描述**：
- 整个项目中几乎没有使用 `javax.validation` 或 `hibernate-validator` 体系的校验注解（如 `@Valid`、`@NotBlank`、`@Size`、`@Pattern` 等）。
- 用户注册、创建/修改短链接等核心接口的必填项校验缺失，大量依赖于 `if-else` 或等请求打穿到底层数据库后由于约束/空指针失败。
**风险**：
- **安全风险**：恶意的 XSS 载荷或超长文本能够轻易穿透到服务层甚至存入数据库；SQL 注入探针虽然由于 MyBatis-Plus 的防注入特性被拦下大部分，但依然不安全。
- **稳定性风险**：大量非法的 API 请求会消耗系统资源（连接池、线程池），本应在 Controller 层被拦住。

### 5. 布隆过滤器设计与生命周期的冲突
**位置**：`ShortLinkServiceImpl.java`
**缺陷描述**：
- 为了防止短链接生成冲突和缓存穿透，系统到处使用了 Redisson 的布隆过滤器（如 `shortUriCreateCachePenetrationBloomFilter`）。
- 但是，布隆过滤器的算法特性决定了它**不支持物理删除**（除非使用带计数器的复杂变种）。
- 项目里的 `updateShortLink` 的先删后插逻辑，或者假如有回收站的彻底销毁逻辑，都**无法**将废弃的 `fullShortUrl` 从布隆过滤器中剔除。长此以往（特别是一些无效跳转被恶意扫描后存入布隆过滤器），其误判率会伴随时间严重上升，导致误把合法链接判定为不存在，或阻碍部分可复用的 Hash 挂载。

### 6. 登录状态管理的脆弱性
**位置**：`UserServiceImpl.java` 的 `login` 方法
**缺陷描述**：
- 登录逻辑中，判断到用户已登录后，直接通过 Redis 延期旧 token：`stringRedisTemplate.expire(USER_LOGIN_KEY + requestParam.getUsername(), 30L, TimeUnit.MINUTES);`。
- 这种设计没有实现完整的 Session 生命周期与终端类型挂钩。如果该用户在 A 电脑登录，再去 B 电脑登录，由于取的是同一个 Redis Key 下唯一的 Token，可能会导致并发冲突，或者 A 电脑被莫名踢下线（且踢线无明确提示）；或者如果不允许重复登录，逻辑表现为“必须等半小时自动过期后才能换设备”。此模块缺少多端在线、同端互踢的正规 JWT / Session 方案。

---

## 三、 改进建议

1. **引入密码哈希算法**：使用 `BCryptPasswordEncoder` 强制在注册时加密，密码字段永远不返回给前端。
2. **鉴权下沉或补充网关密钥**：
   - 内部微服务在 HTTP 通信时不仅传递 Header，还可以附带网关签发的内部 JWT 或秘钥，微服务过滤器校验内网防伪签名。
   - 严格在云厂商安全组/K8s中配置端口封闭规则。
3. **补充接口越权拦截**：在任何涉及用户私密信息的 Service 方法最前面，统一检查 `if (!Objects.equals(username, UserContext.getUsername())) throw new ClientException("无权访问");`
4. **全面接入 Validator**：对于 `UserRegisterReqDTO`、`ShortLinkCreateReqDTO`，强制限定字符格式、长度，并在 Controller 添加 `@Validated`。
