# Shortlink 项目远程调用剖析

本文档详细介绍了 Shortlink 项目中哪些地方用到了微服务之间的远程调用（Remote Procedure Call），以及其底层实现原理和上下文透传机制。

## 1. 项目中哪里用到了远程调用？

在 Shortlink 的架构设计中，业务被划分为了多个独立运行的微服务模块，最核心的两个是：
*   **Admin 模块 (short-link-admin)**：负责用户管理、短链接的分组管理等偏“后台控制台”的逻辑。
*   **Project 模块 (short-link-project)**：真正负责短链接的创建、跳转路由、数据监控统计、回收站管理等“核心业务”逻辑。

**使用场景：从 Admin 模块调用 Project 模块**
当用户在前端控制台点击“创建短链接”、“查看监控数据”或“移入回收站”时，请求首先会打到 Admin 模块的 Controller 层。但 Admin 模块并没有这些核心业务的数据库表，所以它必须**远程调用** Project 模块提供的接口来完成实际操作。

**核心代码位置**：
在 Admin 模块的 `com.nageoffer.shortlink.admin.remote.ShortLinkActualRemoteService` 接口中，集中定义了所有对 Project 模块的远程调用。例如：
*   `createShortLink` (创建短链)
*   `pageShortLink` (分页查询短链)
*   `oneShortLinkStats` (查询监控数据)
*   `saveRecycleBin` (保存至回收站) 等等几十个接口。

## 2. 远程调用的实现原理：Spring Cloud OpenFeign

项目使用了 Spring Cloud 生态中的 **OpenFeign** 组件来实现微服务间的 HTTP 远程调用。

### 2.1 什么是 OpenFeign？
OpenFeign 是一个声明式的 Web Service 客户端。它的原理非常精妙：你只需要写一个普通的 Java 接口，并在上面打上各种 Spring MVC 的注解（比如 `@GetMapping`, `@PostMapping` 等），剩下的构建 HTTP 请求、发起网络调用、解析 JSON 返回值等繁琐工作，Feign 会在底层帮你全部搞定。

### 2.2 工作原理演示
在 `ShortLinkActualRemoteService` 中是这样写的：
```java
@FeignClient(value = "short-link-project", url = "${aggregation.remote-url:}", configuration = OpenFeignConfiguration.class)
public interface ShortLinkActualRemoteService {
    @PostMapping("/api/short-link/v1/create")
    Result<ShortLinkCreateRespDTO> createShortLink(@RequestBody ShortLinkCreateReqDTO requestParam);
    // ...
}
```

*   **@FeignClient 注解**：告诉 Spring 容器，这是一个 Feign 客户端。`value = "short-link-project"` 指定了要调用的目标微服务名称。如果有注册中心（如 Nacos），Feign 会自动从注册中心去拉取目标微服务的真实 IP 和端口进行负载均衡调用；如果没有注册中心，也可以通过 `url` 属性直连。
*   **方法声明**：`@PostMapping("/api/short-link/v1/create")` 和 `@RequestBody` 严格对应了目标微服务（Project 模块）中真实 Controller 的请求路径和参数格式。
*   **动态代理**：当你在 Admin 模块的 Service 里注入 `ShortLinkActualRemoteService` 并调用它的方法时，实际上执行的是 Feign 在系统启动时为你生成的**动态代理类**。这个代理类会将你的 Java 方法调用转换为一次真实的 HTTP POST 网络请求，发送给 `short-link-project` 服务，并将拿到的 JSON 结果反序列化回 `Result<ShortLinkCreateRespDTO>` 对象。对于开发者来说，就像调用本地的一个普通方法一样丝滑。

## 3. 核心大坑与解决：用户信息如何跨服务透传？

微服务架构中有一个经典的痛点：**上下文丢失**。

我们之前提到，网关校验完 Token 后，把用户信息放到了 Request Header 里传给了 Admin 模块，Admin 模块通过 `UserTransmitFilter` 把用户信息存到了当前线程的 `ThreadLocal` 里。
**但是，当 Admin 模块通过 OpenFeign 向 Project 模块发起一次全新的 HTTP 远程调用时，这个新的 HTTP 请求默认是不会带上之前的 Header 的！** 这会导致 Project 模块收不到操作人的身份信息，报出未登录或无权限的错误。

### 3.1 解决方案：OpenFeign 的请求拦截器
为了解决这个问题，项目专门配置了 `OpenFeignConfiguration`。

```java
@Configuration
public class OpenFeignConfiguration {
    @Bean
    public RequestInterceptor requestInterceptor() {
        return template -> {
            String username = UserContext.getUsername();
            String realName = UserContext.getRealName();
            template.header("username", username != null ? URLEncoder.encode(username, StandardCharsets.UTF_8) : null);
            template.header("userId", UserContext.getUserId());
            template.header("realName", realName != null ? URLEncoder.encode(realName, StandardCharsets.UTF_8) : null);
        };
    }
}
```

### 3.2 拦截器工作原理解析
1.  **挂载配置**：在 `@FeignClient` 注解中，通过 `configuration = OpenFeignConfiguration.class` 将这个配置类绑定到了远程调用服务上。
2.  **执行时机**：每当 Feign 准备好了一个 HTTP 请求模板（`template`），在**真正发送到网络之前**，都会先经过这层 `RequestInterceptor` 拦截器。
3.  **提取与塞入（透传）**：
    *   拦截器去 `UserContext`（其实就是当前线程的 ThreadLocal）中取出之前保存的当前登录人的 `userId`、`username`、`realName`。
    *   通过 `template.header(...)`，把这些用户信息作为全新的 Header 键值对，**硬塞进即将发出去的 Feign HTTP 请求头中**。
4.  **接收处理**：Project 模块作为被调用方，同样配置了 `UserTransmitFilter` 拦截器。当它接收到 Admin 模块发来的 Feign HTTP 请求时，会像处理普通前端请求一样，从 Header 里把这三个用户信息读出来并存入自己的 ThreadLocal 中，从而完成了用户身份在各个微服务之间的无缝漫游。

这就是 Shortlink 项目中远程调用的运用及其背后最重要的用户身份透传原理。
