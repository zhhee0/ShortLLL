# ShortLink 项目分表策略分析

## 一、技术选型

项目使用 **Apache ShardingSphere-JDBC 5.3.2** 作为分表中间件，以 JDBC 驱动的方式嵌入应用中，无需额外部署中间件服务。配置文件位于各模块的 `resources/shardingsphere-config-*.yaml`。

---

## 二、分表概览

项目共对 **4 张表** 进行了分表，每张表均拆分为 **16 个子表**（编号 0~15），使用 **HASH_MOD**（哈希取模）算法进行路由。

| 逻辑表名 | 物理表 | 分片键 | 所属模块 |
|-----------|--------|--------|----------|
| `t_user` | `t_user_0` ~ `t_user_15` | `username` | admin |
| `t_group` | `t_group_0` ~ `t_group_15` | `username` | admin / project |
| `t_link` | `t_link_0` ~ `t_link_15` | `gid`（分组标识） | project |
| `t_link_goto` | `t_link_goto_0` ~ `t_link_goto_15` | `full_short_url`（完整短链接） | project |

**分片算法**：统一使用 `HASH_MOD`，即对分片键的哈希值进行取模（`hash(shardingKey) % 16`），决定数据落入哪个子表。

---

## 三、各表分片键选择及原因

### 1. `t_user` 表 — 分片键：`username`

**表结构关键字段**：`id`、`username`、`password`、`real_name`、`phone`、`mail`

**为什么选 `username`**：

- **业务唯一标识**：`username` 是用户的业务唯一标识，系统中用户登录、查询个人信息等操作全部以 `username` 为条件，选它作为分片键可以保证这些高频查询都能**精确路由到单表**，避免跨表扫描。
- **天然离散性好**：用户名是字符串类型，哈希后分布均匀，不会出现数据倾斜。
- **不可变性**：用户名一旦注册不会修改，避免了分片键变更导致的数据迁移问题。

---

### 2. `t_group` 表 — 分片键：`username`

**表结构关键字段**：`id`、`gid`（分组标识）、`name`（分组名称）、`username`（创建者）、`sort_order`

**为什么选 `username` 而不是 `gid`**：

- **查询场景驱动**：分组的核心查询场景是"查询某个用户的所有分组"，即 `WHERE username = ?`。选 `username` 作为分片键，这类查询可以**精确路由到单表**。
- **同用户数据同表**：同一用户创建的所有分组落在同一个子表中，方便进行排序、分页等操作，无需跨表聚合。
- **与 `t_user` 表同步**：`t_user` 和 `t_group` 都以 `username` 为分片键，使得同一个用户的用户信息和分组信息落在相同编号的子表中，虽然是不同逻辑表（ShardingSphere 中叫绑定表），但利于后续可能的关联查询优化。

---

### 3. `t_link` 表 — 分片键：`gid`

**表结构关键字段**：`id`、`domain`、`short_uri`、`full_short_url`、`origin_url`、`gid`（分组标识）、`click_num` 等

**为什么选 `gid`**：

- **管理侧查询场景**：短链接的管理操作（分页查询、统计数量）几乎都是按分组维度进行的，如"查询某分组下的所有短链接"（`WHERE gid = ?`）。用 `gid` 作为分片键，这类查询可以精确路由。
- **同分组数据聚合**：同一分组下的短链接落在同一子表，分页查询和 `count` 统计都在单表内完成，性能有保障。
- **更新操作的考量**：当用户修改短链接所属分组时（变更 `gid`），由于分片键发生变化，代码采用了**先删后插**的策略（见 `ShortLinkServiceImpl.updateShortLink()` 中 gid 不同的分支），从旧表删除记录再插入到新的目标表。

**为什么不选 `full_short_url`**：

> 如果选 `full_short_url` 作为 `t_link` 的分片键，虽然跳转查询可以直接路由，但管理侧"查询某分组下所有短链接"这类高频操作就会变成**全表扫描**（因为同一分组的短链接会散落在不同子表中），这在业务上是不可接受的。

---

### 4. `t_link_goto` 表 — 分片键：`full_short_url`

**表结构关键字段**：`id`、`gid`、`full_short_url`

**这张表的作用**：

`t_link_goto` 是一张**路由中间表**，专门用于解决短链接跳转场景下的分表路由问题。它的核心作用是：**通过 `full_short_url` 查到对应的 `gid`，再用 `gid` 去 `t_link` 表查询完整的短链接信息**。

**为什么需要这张表**：

- 短链接跳转（`restoreUrl`）时，请求只携带短链接 URL，不知道 `gid` 是什么。
- `t_link` 表的分片键是 `gid`，如果不知道 `gid` 就无法精确路由，只能扫描全部 16 张子表。
- 因此引入 `t_link_goto` 表，以 `full_short_url` 为分片键，先查这张表拿到 `gid`，再用 `gid` 精确查 `t_link` 表。

**为什么选 `full_short_url`**：

- **跳转场景唯一入参**：用户访问短链接时，唯一已知的信息就是 `full_short_url`，以它为分片键可以精确路由到单表。
- **读多写少**：这张表在创建短链接时写入一次，之后每次跳转都会读取，是典型的读多写少场景，用 `full_short_url` 作为分片键能保证高频读操作的性能。

---

## 四、跳转流程中的两次路由示意

```
用户访问短链接 (full_short_url)
        │
        ▼
  ┌─────────────────┐
  │  Redis 缓存查询   │ ── 命中 ──> 直接 302 跳转
  └─────────────────┘
        │ 未命中
        ▼
  ┌─────────────────────────────────────┐
  │ 第1次路由：查 t_link_goto 表          │
  │ 分片键: full_short_url               │
  │ 目的: 获取 gid                       │
  └─────────────────────────────────────┘
        │
        ▼
  ┌─────────────────────────────────────┐
  │ 第2次路由：查 t_link 表               │
  │ 分片键: gid + full_short_url         │
  │ 目的: 获取 origin_url（原始链接）      │
  └─────────────────────────────────────┘
        │
        ▼
    302 跳转到原始链接
```

## 五、总结

| 设计决策 | 说明 |
|---------|------|
| **分片算法** | 统一使用 HASH_MOD，简单高效，数据分布均匀 |
| **分片数量** | 统一 16 张子表，预留了足够的扩展空间 |
| **分片键选择核心原则** | 以**最高频查询条件**为分片键，确保精确路由，避免跨表扫描 |
| **路由中间表设计** | 引入 `t_link_goto` 表解决"跳转场景不知道 `gid`"的问题，是一种典型的**空间换时间**策略 |
| **分组变更处理** | 修改短链接分组时采用先删后插策略，因为分片键 `gid` 变更会导致数据需要迁移到新的子表 |

## 六、统计表为什么没有分表？（值得思考的问题）

项目中与统计相关的表有 **7 张**，数据量理论上远超 `t_link`（每条短链接每次被访问都可能产生多条统计记录），但**全部没有做分表**：

| 统计表 | 用途 | 数据量级 |
|--------|------|---------|
| `t_link_access_stats` | 基础访问统计（PV/UV/UIP，按小时聚合） | 每链接 × 每小时 |
| `t_link_access_logs` | 访问明细日志（每次访问一条） | **最大**，每次点击都记录 |
| `t_link_stats_today` | 今日统计 | 每链接 × 每天 |
| `t_link_locale_stats` | 地区统计 | 每链接 × 每地区 |
| `t_link_os_stats` | 操作系统统计 | 每链接 × 每OS |
| `t_link_browser_stats` | 浏览器统计 | 每链接 × 每浏览器 |
| `t_link_device_stats` | 设备统计 | 每链接 × 每设备类型 |
| `t_link_network_stats` | 网络统计 | 每链接 × 每网络类型 |

### 这是项目的一个不足 / 简化处理

**没有分表的可能原因**：

1. **项目定位**：这是一个教学/演示项目，统计模块的分表会大幅增加复杂度（分片键选择更纠结、跨表聚合更复杂），作者可能有意简化。
2. **分片键选择困难**：统计表的查询场景比较复杂——既要按 `full_short_url` 查单链接统计，又要按 `gid` 查分组维度统计，还要按日期范围聚合。无论选哪个字段做分片键，都会有部分查询场景需要跨表扫描。

### 如果要对统计表做分表，应该怎么做？

**推荐方案：以 `full_short_url` 为分片键**

- **理由**：统计的最小粒度是单个短链接，绝大多数查询都是"查某条短链接的统计数据"。
- **分组维度查询**：可通过应用层先查出分组下所有 `full_short_url`，再逐个查询后聚合（或通过路由中间表做映射）。
- **`t_link_access_logs`（访问日志表）**：数据量最大，最应该优先分表，甚至可以考虑按时间分区 + 按 `full_short_url` 分表的组合策略。

**另一种思路：统计数据不放 MySQL**

在实际生产中，更常见的做法是将高频写入的统计数据存入 **ClickHouse、Elasticsearch** 等专用 OLAP 引擎，MySQL 只存核心业务数据。项目中实际上已经引入了部分这样的思路（用 Redis 做实时计数缓冲，通过消息队列异步写入数据库），但最终落地还是单表 MySQL。

---

## 七、统计表都没分表，`t_link` 和 `t_link_goto` 还有必要分表吗？

**结论：有必要，但原因不完全是数据量。**

### `t_link_goto` —— 最有必要分表

- 它处于**系统最高频路径**上：每一次短链接跳转都可能查它（Redis 缓存未命中时）。
- 分表的核心目的不是"数据量大"，而是**分散读压力**，避免高并发跳转时单表成为瓶颈。
- 一条热门短链接每秒可能被访问成千上万次，即使有 Redis 挡住大部分流量，穿透到数据库的查询也需要快速响应。

### `t_link` —— 有必要，但没那么紧迫

- 短链接数量到千万/亿级时，单表查询、索引维护都会变慢，分表是合理的。
- 按 `gid` 分表后，管理侧的分页查询和分组统计可以在单表内完成，性能有保障。

### 但项目设计确实存在不一致

> **核心矛盾**：如果系统的短链接量真的大到需要给 `t_link`（数据量 = 短链接数）分表，那么 `t_link_access_logs`（数据量 = 所有短链接的总点击次数）早就先撑爆了。

| 表 | 数据量量级 | 是否分表 |
|----|----------|---------|
| `t_link` | = 短链接总数（百万~千万级） | ✅ 已分表 |
| `t_link_goto` | = 短链接总数 | ✅ 已分表 |
| `t_link_access_logs` | = 总点击次数（可能是短链接数的 100~10000 倍） | ❌ 未分表 |
| `t_link_access_stats` | = 短链接数 × 小时数 | ❌ 未分表 |

**如果在面试中被问到这个问题，可以这样回答**：

> "项目对 `t_link` 和 `t_link_goto` 的分表设计是合理的，分别优化了管理查询和跳转查询两个核心场景。但统计表没有分表是一个明显的不足——在真实生产环境中，统计表的数据增长速度远超链接表，应该优先考虑分表或者使用 ClickHouse 等 OLAP 引擎来承接统计数据。"
