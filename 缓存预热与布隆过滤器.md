# 缓存预热与布隆过滤器初始化机制详解

---

## 一、布隆过滤器（Bloom Filter）的初始化

### 1.1 项目中一共有三个布隆过滤器

| 布隆过滤器名称 | 所在模块 | 预期容量 | 误判率 | 作用 |
| :--- | :--- | :--- | :--- | :--- |
| `shortUriCreateCachePenetrationBloomFilter` | Project 模块 | **1 亿** | 0.001 (千分之一) | 防止恶意用户伪造不存在的短链接疯狂请求，穿透缓存直击数据库 |
| `userRegisterCachePenetrationBloomFilter` | Admin 模块 | **1 亿** | 0.001 | 用户注册时快速判断用户名是否已被占用,避免每次注册都查库 |
| `gidRegisterCachePenetrationBloomFilter` | Admin 模块 | **2 亿** | 0.001 | 分组标识(gid)创建时判断是否重复占用 |

### 1.2 初始化方式：`tryInit` + Spring Bean

布隆过滤器的初始化逻辑写在 `RBloomFilterConfiguration` 配置类中，以 `@Bean` 的形式注册到 Spring IoC 容器：

```java
@Bean
public RBloomFilter<String> shortUriCreateCachePenetrationBloomFilter(RedissonClient redissonClient) {
    RBloomFilter<String> cachePenetrationBloomFilter = 
        redissonClient.getBloomFilter("shortUriCreateCachePenetrationBloomFilter");
    cachePenetrationBloomFilter.tryInit(100000000L, 0.001);
    return cachePenetrationBloomFilter;
}
```

### 1.3 `tryInit` 的关键细节（面试高频追问）

> **面试官**："布隆过滤器每次服务重启都会重新初始化吗？之前存进去的数据会丢失吗？"

**你的回答：**
> "不会。这里用的是 **Redisson 提供的分布式布隆过滤器**，底层数据结构（位数组 BitSet）是存储在 **Redis 服务器**里的，不是存在 JVM 内存里。
> 
> `tryInit(expectedInsertions, falseProbability)` 方法的语义是**'尝试初始化'**——如果 Redis 里已经存在同名的布隆过滤器（说明这不是第一次启动），它会直接返回 `false` 并**跳过初始化，复用已有的数据**。只有在 Redis 里没有这个 Key 时（比如第一次部署或者 Redis 被清空），才会真正创建一个新的空布隆过滤器。
> 
> 所以服务重启完全不会导致布隆过滤器数据丢失，这也是选择 Redisson 分布式布隆过滤器而非 Guava 本地布隆过滤器的核心原因之一。"

### 1.4 布隆过滤器的数据是怎么填入的？

布隆过滤器里的数据**不是启动时批量灌入**的，而是在**业务发生时实时写入**的。

**短链接布隆过滤器（`shortUriCreateCachePenetrationBloomFilter`）：**
在 `ShortLinkServiceImpl.createShortLink` 中，每当一条新的短链接被成功创建并插入数据库后，立刻调用：
```java
shortUriCreateCachePenetrationBloomFilter.add(fullShortUrl);
```
把这条短链的完整 URL 加入布隆过滤器。

**用户名布隆过滤器（`userRegisterCachePenetrationBloomFilter`）：**
在 `UserServiceImpl.register` 中，注册成功后调用：
```java
userRegisterCachePenetrationBloomFilter.add(requestParam.getUsername());
```

> **面试话术总结**：布隆过滤器采用的是**"写时同步追加"**策略，而不是批量预加载。每次有新的合法数据产生时，实时向布隆过滤器中追加记录。由于布隆过滤器的数据持久化在 Redis 中，即使应用重启也不会丢失历史数据。

## 二、缓存预热（Cache Warmup）

### 2.1 什么是缓存预热？

缓存预热是指在缓存系统上线或重启后，**提前将热点数据加载到缓存中**，避免上线瞬间所有请求全部打到数据库导致"缓存雪崩"。

### 2.2 本项目的缓存预热策略：创建时即预热（Write-Through）

本项目对短链接缓存采用的是最直接的策略——**在短链接被创建的那一刻，就立即将其写入 Redis 缓存**，而不是等到第一个用户访问时才加载（那叫懒加载/Cache Aside）。

在 `ShortLinkServiceImpl.createShortLink` 中：
```java
// 创建短链成功后，立即将「短链 → 原始链接」的映射关系存入 Redis
stringRedisTemplate.opsForValue().set(
    String.format(GOTO_SHORT_LINK_KEY, fullShortUrl),   // key: goto:nurl.ink/abc123
    requestParam.getOriginUrl(),                         // value: https://www.baidu.com
    LinkUtil.getLinkCacheValidTime(requestParam.getValidDate()),  // TTL: 距离过期还有多少毫秒
    TimeUnit.MILLISECONDS
);
```

这意味着：
1. 短链接一被创建，其对应的跳转目标就已经在 Redis 里了。
2. 第一个用户访问该短链接时，**直接命中 Redis 缓存**，根本不需要去查数据库。
3. 缓存的 TTL（过期时间）被精确设置为距离短链有效时间的剩余毫秒数（通过 `LinkUtil.getLinkCacheValidTime` 计算），如果是永久类型，则使用一个默认的超长过期时间。

### 2.3 缓存未命中时的兜底逻辑（懒加载回源）

如果因为 Redis 的 Key 自然过期了、或者 Redis 被重启导致缓存丢失呢？此时用户访问短链触发 `restoreUrl` 方法，会经历如下过程：

```
用户点击短链 → 查 Redis 缓存 → 未命中
              ↓
     查布隆过滤器 → 不存在 → 直接返回 404（防穿透）
              ↓ 存在
     查 Redis 空值缓存 → 存在("-") → 返回 404（防穿透保护期内）
              ↓ 不存在
     加分布式锁 → 二次查缓存（防止并发重复回源）
              ↓ 仍未命中
     查数据库 → 找到了 → 重新写入 Redis 缓存（重新预热）+ 302 跳转
              ↓ 没找到或已过期
     写入空值到 Redis（30 分钟） → 返回 404
```

**所以本质上是一个"创建时预热 + 回源时重建"的双保险机制。**

### 2.4 面试话术提炼

> "在缓存预热策略上，我采用的是**创建即预热（Write-Through）**的方式。
> 
> 当管理员在后台创建一条新的短链接时，系统在成功落库的同时，会立刻将'短链 → 原链'的映射关系同步写入 Redis，并且根据这条短链的有效期精确计算出缓存的 TTL。这样从创建完成的第一秒起，所有用户访问这条短链都是直接命中 Redis 缓存的，不会打到数据库。
> 
> 对于缓存过期或 Redis 宕机等极端情况，我还设计了一套完整的**回源重建机制**：请求到来时先过布隆过滤器拦截非法短链，再加分布式锁防止并发穿透，最后查库回填缓存。整个流程保证了缓存层在任何场景下都不会成为系统的薄弱点。"

## 三、面试连环问速查表

| 面试官追问 | 你的回答要点 |
| :--- | :--- |
| 布隆过滤器数据存在哪？ | **Redis 里**，不是 JVM 内存。用的 Redisson 分布式布隆过滤器。 |
| 服务重启布隆过滤器会清空吗？ | **不会**。`tryInit` 是幂等的，已存在则跳过。数据持久化在 Redis。 |
| 布隆过滤器的数据什么时候灌进去的？ | **业务发生时实时写入**（`add`），不是批量预加载。 |
| 布隆过滤器误判怎么办？ | 误判只会导致"本不存在的短链"通过了布隆过滤器而多查了一次数据库（查完发现不存在就缓存空值 30 分钟）。**不会漏判**已存在的短链。 |
| 缓存预热是启动时做的吗？ | **不是启动时批量加载**（那样效率极低）。而是**创建短链时就同步写缓存**,属于 Write-Through 模式。 |
| 如果 Redis 挂了重启，缓存全没了怎么办？ | 有兜底：布隆过滤器 → 分布式锁 → 查库回填。把热点数据**按需懒加载重建**回 Redis。 |
