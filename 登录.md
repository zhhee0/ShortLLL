# Shortlink 项目登录流程详解

本文档详细介绍了 Shortlink 项目中用户登录的完整流程，包括前后端的交互、Token 的生成与校验、以及拦截器的工作机制。

## 1. 整体架构与入口

*   **前端触发**: 用户在控制台页面输入账号密码，点击登录。
    *   请求路径: `/api/short-link/admin/v1/user/login`
*   **网关转发**: 请求首先到达网关层 (Gateway)。
    *   网关的白名单机制 (`config.getWhitePathList()`) 会放行登录、注册等基础接口，所以无需 Token 即可透传给后台业务微服务。
*   **后端处理**: Admin 微服务接收请求。
    *   处理类: `UserController.login` -> `UserServiceImpl.login`

## 2. 后端登录核心逻辑 (UserServiceImpl)

当登录请求到达 `UserServiceImpl.login` 方法时，执行以下核心步骤：

### 2.1. 验证用户身份
首先去数据库校验用户名、密码，并且确保该用户未被逻辑删除 (`delFlag = 0`)。
```java
LambdaQueryWrapper<UserDO> queryWrapper = Wrappers.lambdaQuery(UserDO.class)
        .eq(UserDO::getUsername, requestParam.getUsername())
        .eq(UserDO::getPassword, requestParam.getPassword())
        .eq(UserDO::getDelFlag, 0);
UserDO userDO = baseMapper.selectOne(queryWrapper);
if (userDO == null) {
    throw new ClientException("用户不存在");
}
```

### 2.2. 检查是否重复登录（续期机制）
检查 Redis 中是否已经存在该用户的登录 Token。
如果存在，说明用户之前已经登录过且 Token 尚未过期：
*   直接获取旧的 Token。
*   **关键点：将该 Token 在 Redis 中的有效期“续命”至 30 分钟。**
*   返回旧 Token 给前端。
```java
Map<Object, Object> hasLoginMap = stringRedisTemplate.opsForHash()
        .entries(USER_LOGIN_KEY + requestParam.getUsername());
if (CollUtil.isNotEmpty(hasLoginMap)) {
    stringRedisTemplate.expire(USER_LOGIN_KEY + requestParam.getUsername(), 30L, TimeUnit.MINUTES);
    String token = hasLoginMap.keySet().stream().findFirst().map(Object::toString).orElseThrow(...);
    return new UserLoginRespDTO(token);
}
```

### 2.3. 生成新 Token 并保存
如果用户是全新登录（Redis 中没有记录）：
*   生成一个全局唯一的 UUID 字符串作为鉴权 Token。
*   使用 Hash 结构将用户信息存入 Redis：
    *   Key: `short-link:login:用户名`
    *   Hash Key: `UUID (Token)`
    *   Hash Value: 包含用户完整信息的 JSON 序列化字符串。
*   **设置过期时间：30 分钟**。
*   将新 Token 返回给前端。
```java
String uuid = UUID.randomUUID().toString();
stringRedisTemplate.opsForHash().put(USER_LOGIN_KEY + requestParam.getUsername(), uuid, JSON.toJSONString(userDO));
stringRedisTemplate.expire(USER_LOGIN_KEY + requestParam.getUsername(), 30L, TimeUnit.MINUTES);
return new UserLoginRespDTO(uuid);
```

## 3. 前端 Token 处理 (auth.js)

前端在收到接口返回的 Token 成功后，会将其保存在 Cookie 中。

```javascript
// core/auth.js
export function setToken(token) {
  return Cookies.set(TokenKey, token) // 未设置 expires 属性
}
```

*   **特点**: 由于没有显式设置 `expires`（过期时间），此时的 Cookie 通常被视为**会话 Cookie (Session Cookie)**。这意味着只要浏览器进程不被彻底关闭（或人为清除缓存），这串 Token 就会一直存在于本地。（这就是为什么有时表现出“几天都不用重新登录”假象的其中一个原因，具体配合后端的续期机制）。

## 4. 后续请求的鉴权拦截与用户信息透传

当用户登录成功后，接下来的业务请求都必须携带 Token。这里分为两个阶段：**网关层的鉴权与透传**，以及**微服务层的接收与上下文保存**。

### 4.1. 网关层 (Gateway)：拦截与向下透传
拦截工作统一由网关层的 `TokenValidateGatewayFilterFactory` 处理。
1.  **非白名单校验**: 判断请求路径是否在白名单外。
2.  **获取 Header 信息**: 从 HTTP Header 中提取 `username` 和 `token`。
3.  **Redis 校验**: 拿着 `username` 和 `token` 去 Redis 中查询对应的用户信息副本。
4.  **请求透传与信息注入（关键所在）**: 如果校验成功：
    *   拦截器会将从 Redis 解析出的用户信息（`userId`, `username`, `realName`）重新**写入到当前 HTTP Request 的 Header 中**。
    *   因为有些中文字符（如真实姓名），写入 Header 前还会经过 `URLEncoder.encode` 编码处理。
    *   改写后的 Header 会跟随着 HTTP 请求正式转发给最终的后台微服务（比如 Admin 或 Project 模块）。

### 4.2. 微服务层 (Admin/Project)：接收与上下文绑定（UserTransmitFilter）
当带有用户信息的 HTTP 请求到达具体的微服务时，由于微服务内部执行业务逻辑时随时可能需要获取“当前操作人是谁”，如果每次都从 Request Header 里拿会非常繁琐且侵入性强。

所以系统中引入了 `UserTransmitFilter`（用户信息传输过滤器）：
1.  **解析 Header**: 该 Filter 会拦截进入微服务的所有请求，读取刚才网关塞进 Header 里的 `userId`，`username` 和 `realName`。并且对编码过的字符使用 `URLDecoder.decode` 进行解码。
2.  **存入 ThreadLocal**: 将这些信息封装成 `UserInfoDTO`，然后调用 `UserContext.setUser(userInfoDTO)`。
    *   `UserContext` 的底层是基于 **ThreadLocal（线程本地变量）** 实现的。
    *   Spring Boot 处理每一个 HTTP 请求都会分配一个独立的线程，把用户信息存在当前线程的 ThreadLocal 里，就能保证在这个请求的完整生命周期内，任何处于该线程深处的 Service 层或 Mapper 层代码，只需要调用 `UserContext.getUsername()` 就能随时随地、极其方便地拿到当前登录人的信息。
3.  **清理释放**: 当这个请求处理完毕并准备响应时（位于 `finally` 块），Filter 会强制调用 `UserContext.removeUser()` 清理掉当前线程绑定的用户信息。这步非常关键，因为 Tomcat 内部的线程池是复用的，如果不清理，下一个无关请求分到这个脏线程时，可能会发生严重的越权串车 Bug。


### 4.2. 拦截器存在的不足
值得注意的是，虽然请求被拦截校验，但当前的网关拦截器**并未在用户每次请求有效接口时主动对 Token 进行时间顺延（续期）**。
这意味着，只要你一直在线但不主动触发 `/login` 接口，你的 Token 理论上必定会在 30 分钟后迎来真正的“硬过期”。

## 5. 总结

1.  **登录成功**：生成随机 UUID Token。
2.  **后端存储**：Redis 保存 Token 及用户信息，统一设置 30 分钟有效期。
3.  **前端存储**：Cookie 保存 Token（通常为基于会话的时效）。
4.  **鉴权机制**：网关过滤器拦截非白名单请求，验证 Redis 中 Token 存在且匹配。并在验证通过后将用户信息塞入 Header 传递给下游。
5.  **续期机制**：目前系统只有在重复调用 `/login` 接口时才会给处于有效期内的老 Token 重新续命 30 分钟。网关层并不负责日常请求访问的续期动作。
